# SEO 동적 렌더링 재점검 및 SSG 방향 결정 인수인계 문서

작성일: 2026-06-23

## 1. 목표

네이버 검색 노출 관점에서 가장 치명적인 문제로 지목된 "상품 콘텐츠가 JS
렌더링 후에만 보이는 구조"를 해결한다. 방향은 **방법 A(빌드타임 정적
생성, SSG)** 로 확정되었고, 이번 세션은 방향 확정 직전까지의 재점검만
진행했다. 실제 SSG 구현은 다음 세션에서 시작한다.

## 2. 현재 상태

**배경**: 이 작업은 2026-06-19 세션에서 처음 점검되었고, 이후 6/19~6/22
사이 다른 작업(코인 스펙 검증, 한글화 등)에 밀려 진행되지 않은 채
방치되어 있었다. 이번 세션에서 현재 코드 기준으로 재점검했다.

**확인 완료 — 그 사이(6/19→6/23) 이미 해결된 것** (다른 세션 작업分):
- og:태그 — index/coins/gold-price/contact/coin-detail 5개 페이지 적용 완료
- canonical 태그 — 위 5개 페이지 적용 완료
- favicon — 위 5개 페이지 적용 완료
- index.html, coin-detail.html에 JSON-LD 스크립트 추가 시도됨

**확인 완료 — 핵심 문제는 6/19 점검 시점과 동일하게 그대로 남아있음**:
1. `index.html`, `coins.html` — 초기 HTML에 상품 콘텐츠 없음.
   `.products-loading` + 로딩 스피너만 있고, 실제 상품명·가격은 구글시트
   gviz API를 fetch한 뒤 JS가 DOM에 그려넣음 (`index.html` 196~198행,
   `coins.html` 67~69행 직접 확인).
2. `coin-detail.html` — 여전히 `?id=...` 쿼리파라미터로 동작
   (539행 `URLSearchParams`). 콘텐츠는 `innerHTML`로 JS가 채움 (290,
   510, 534행).
3. `coin-detail.html`의 JSON-LD도 **JS가 `document.head`에 동적으로
   `appendChild`** 하는 방식(242~245행) — 정적 크롤러에게는 안 보이는
   문제가 메타태그 레이어에서도 그대로 반복되고 있음.
4. `coin-detail.html`의 `canonical`이 **모든 코인 상세페이지에서
   동일하게 `coins.html`을 가리킴** (8행) — 코인별 고유 URL이 아님.
5. `coin-detail.html`의 `og:image`/`og:title`도 **메이플 코인으로
   하드코딩 고정** (15행) — 어떤 코인을 보든 카카오톡/SNS 공유 시 같은
   미리보기가 뜸.
6. `sitemap.xml` — 6/19 그대로, 5개 URL만 등록(`/`, `coins.html`,
   `coin-detail.html` 단일, `gold-price.html`, `contact.html`). 코인
   19개 개별 URL 없음.

**새로 발견** (이번 세션):
- `mypage.html`은 og/canonical/favicon 전부 0개 — 6/19 점검 범위(6개
  페이지) 중 메타태그 보강 작업에서 누락된 것으로 보임.

**결론**: 메타태그 레이어는 부분적으로 보강됐지만, "네이버봇이 실제로
무엇을 보는가"라는 근본 문제(JS 의존 렌더링)는 전혀 해결되지 않음.

## 3. 진행 단계

방향은 **방법 A(GitHub Actions 기반 빌드타임 정적 생성)** 로 확정.
다음 세션에서 아래 순서로 진행한다.

1. (필수) GitHub Actions 워크플로우 설계
   - 트리거: 주기적(예: 1일 1회) + 수동 실행(workflow_dispatch) 둘 다 고려
   - 구글 시트 gviz API를 Actions 내에서 fetch해 상품 데이터 확보
   - `coin-descriptions.js`(정적 스펙)와 시트 데이터(가격 등 동적 값)를
     결합해 정적 HTML 생성
2. (필수) 생성 대상 범위 결정
   - 목록 페이지(index.html 상품 그리드, coins.html 상품 그리드)에
     정적 상품 카드 HTML을 미리 박아넣는 방식인지,
   - 코인별 개별 정적 페이지(`coin-{slug}.html` 등)를 새로 만드는
     방식인지, 혹은 둘 다인지 — 다음 세션 시작 시 사용자와 먼저 확정
3. (필수) 코인별 canonical/og:title/og:image를 코인마다 다르게 만드는
   방법 결정 (정적 페이지 분리 시 자동 해결되지만, 현재처럼 쿼리파라미터
   유지 시에는 JS로 `<head>`를 고쳐도 크롤러가 못 보므로 별도 전략 필요)
4. (필수) sitemap.xml에 코인 19개 개별 URL을 어떻게 반영할지 결정
   (정적 페이지 생성 시 Actions가 sitemap도 같이 자동 생성하는 것이
   유지보수 면에서 유리해 보임 — 다음 세션에서 사용자와 논의)
5. (선택) `mypage.html` 메타태그 누락분 보강 — 이번 SSG 작업과 별개로
   가볍게 처리 가능, 우선순위는 낮음

## 5. 참고 사항

- **이 작업의 출발점**: 2026-06-19 세션에서 처음 전체 SEO 점검을 했고,
  그때 이미 "JS 렌더링 후에만 상품이 보임"을 가장 치명적인 문제로
  지목했었다. 그 뒤 다른 작업(코인 스펙 검증 등)에 우선순위가 밀려
  6/19~6/22 사이 한 번도 다시 다뤄지지 않았다. 이번 세션은 그 사이
  메타태그가 일부 보강됐는지 재확인하는 데 집중했고, 실제 SSG 구현은
  아직 시작 전이다.
- **방법 A를 고른 이유**: GitHub Pages는 정적 호스팅이라 진짜 서버
  사이드 렌더링은 불가능. 방법 B(초기 HTML 일부 하드코딩)는 시트
  변경 시 수동 동기화 부담과 가격 불일치 위험이 있어 제외. 방법
  C(상세페이지만 정적 분리)는 효과가 상세페이지에 한정되므로, 목록
  페이지까지 포함하는 A가 최종 채택됨.
- **JSON-LD/canonical/og가 "추가는 됐지만 효과가 없는" 이유**: 단순히
  태그가 HTML에 있는지 여부만으로는 부족하다. `coin-detail.html`의
  JSON-LD는 JS가 런타임에 `document.head.appendChild`로 주입하므로,
  정적 HTML만 보는 크롤러 입장에서는 애초에 존재하지 않는 것과
  같다. 다음 세션에서 SSG로 전환할 때, "태그가 있다/없다"가 아니라
  "정적 HTML 응답에 처음부터 포함되어 있는가"를 기준으로 검증해야
  한다.
- **canonical/og가 코인마다 고정값인 이유**: 현재 `coin-detail.html`은
  쿼리파라미터(`?id=...`) 하나로 모든 코인을 처리하는 단일 HTML
  파일이라, `<head>`의 canonical/og 태그는 코인과 무관하게 정적으로
  하드코딩되어 있다(전부 메이플 코인 기준). SSG로 코인별 개별 정적
  페이지를 만들면 이 문제는 자동으로 해결된다.
- **GitHub PAT**: 이전 핸드오프(`2026-06-22-2019`) 기준 만료일
  2026-06-26 오후 5:34 KST로 기록되어 있었음. 다음 세션에서 push가
  필요하면 만료 여부 먼저 확인할 것.
- **앞면/뒷면 기준**: 주제 그림(동물·건물 등) = 앞면(`.png`), 인물
  초상·국가 문장 = 뒷면(`-back.png`)
- **스펙 필드 구조**: `year`, `mint`, `country`, `purity`, `weight`,
  `diameter`, `thickness`(선택), `condition`, `verified`
- **신규 상품 추가/삭제 시**: `coin-descriptions.js` + `products.js`의
  IMAGE_MAP + 시트 행 3곳을 함께 관리 (SSG 도입 시 이 목록에 "생성
  스크립트"도 추가될 가능성 있음 — 다음 세션에서 반영)
