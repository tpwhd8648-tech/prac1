# SSG 1~2단계 진행(코인 정적 페이지 자동 생성) 인수인계 문서

작성일: 2026-06-23

## 1. 목표

네이버 검색 노출의 핵심 블로커였던 "상품 콘텐츠가 JS 렌더링 후에만 보이는
구조"를 해결하기 위해 방법 A(빌드타임 정적 생성, SSG)를 구현한다. 이번
세션은 합의한 3단계(1. 템플릿 1개 손작업 → 2. GitHub Actions로 전체 자동화
→ 3. coins.html 링크 교체) 중 1, 2단계를 완료했다.

## 2. 현재 상태

**완료 (push 반영 확인)**

- **1단계 — `coin-maple-2026.html`** (커밋 `66f6089`): 메이플 코인 1개를
  손으로 작성한 정적 템플릿. 스펙·설명·canonical/og/JSON-LD는 정적 HTML에
  직접 포함, 가격·재고만 기존과 동일하게 브라우저에서 시트 fetch 후 JS로
  갱신. `coin-detail.html`과 화면상 거의 동일하게 보이도록 CSS/인터랙션
  (갤러리 줌, 탭 전환)을 그대로 복제함.
- **2단계 — 자동 생성 스크립트 + GitHub Actions** (커밋 `508e74d`):
  - `scripts/generate-coin-pages.js`: `coin-descriptions.js`(스펙) +
    `products.js`의 `IMAGE_MAP`(이미지 파일명) + 구글 시트(상품 정식명만)를
    키워드로 매칭해 코인별 `coin-{slug}.html`과 `sitemap.xml`을 생성한다.
    슬러그는 `IMAGE_MAP`의 `file` 값에서 `products-`/`.png`를 떼어 만듦
    (예: `products-maple-2026.png` → `maple-2026`).
  - `.github/workflows/generate-coin-pages.yml`: 트리거 3종
    (`coin-descriptions.js`/`products.js`/생성스크립트 수정 후 push,
    매일 1회 스케줄(KST 00:00), 수동 `workflow_dispatch`). 변경 발생 시
    `github-actions[bot]` 명의로 자동 커밋&푸시.

**검증 완료 (확인 방법 명시)**

- 생성 스크립트: 구글 시트 fetch 부분을 mock으로 교체한 테스트
  드라이버로 전체 파이프라인(20개 키워드 중 18개 매칭 성공, 2개는 의도적
  누락시켜 skip 로직 확인) 실행 — 18개 파일 전부 `node -c`로 JS 문법 검증,
  Python `json.loads`로 JSON-LD 문법 검증 통과.
- `vm` 컨텍스트 버그 1개 발견 후 수정: `vm.runInContext`로 실행한
  `const`는 sandbox 객체에 직접 안 붙으므로, 코드 끝에
  `this.__EXPORTED__ = X;`를 덧붙여 명시적으로 꺼내는 방식으로 고침.
- 손작업 1단계 템플릿과 스크립트 자동생성 버전을 `diff`로 비교해
  스펙 테이블·JS 로직(가격 갱신, 갤러리, 줌)이 동일함을 확인. title/
  JSON-LD name 차이는 테스트용 가짜 시트명과 실제 시트명이 달라서
  생긴 것으로, 실제 운영 시트로 돌리면 해당 부분도 정확해짐.
- 실제 구글 시트 fetch는 이 작업 환경(Claude 컨테이너)의 네트워크 허용
  목록에 `docs.google.com`이 없어 직접 실행 불가 — **GitHub Actions
  환경에서 처음 실제로 돌려보는 것이 이번 작업의 첫 실전 테스트**임.
  아직 미확인.

**이번 세션에서 함께 해결된 것** (1·2단계 작업으로 자동 해결):
- 코인별 canonical/og:title/og:image 분리 (코인마다 정적 페이지라 자동 해결)
- JSON-LD를 정적 HTML에 직접 포함 (JS `appendChild` 방식 탈피)
- sitemap.xml에 코인 19개(생성 스크립트 기준 시트 매칭 시 18~20개) 개별
  URL 자동 포함

**PAT 토큰 트러블슈팅 (이번 세션에서 겪은 시행착오, 다음에 참고)**:
- 일반 Fine-grained PAT(Contents Read/Write만 있던 토큰)로
  `.github/workflows/*.yml` 파일을 push하면 GitHub이
  `refusing to allow a Personal Access Token to create or update workflow
  ... without 'workflow' scope`로 거부함 — Actions를 통한 권한 상승을
  막는 GitHub 보안 정책. PAT를 다시 보내도 권한이 그대로면 동일하게 거부됨.
- Fine-grained PAT는 **발급 후 권한 목록 자체를 수정할 수 없음** — 이름/
  만료일만 Edit 가능, 권한(permissions) 변경은 새 토큰을 처음부터 다시
  발급해야 함.
- 해결: Repository permissions에 `Workflows: Read and write`를 포함해
  새로 발급 → 정상 push 성공 (커밋 `508e74d`).

## 5. 참고 사항

- **가격/재고를 정적 파일에 안 박은 이유**: 구글 시트가 "진실 소스"라
  가격·재고가 매일 바뀔 수 있는데, 정적 HTML에 가격을 박아두면 시트와
  어긋날 위험이 있다. 그래서 SSG는 "껍데기(스펙+SEO 메타)만 정적,
  가격/재고는 그대로 브라우저에서 실시간 fetch" 원칙으로 설계함. 이
  원칙은 1단계 손작업 템플릿과 2단계 자동생성 스크립트 모두에 동일하게
  적용됨.
- **상품 정식명(name)은 시트에서만 가져옴**: `coin-descriptions.js`에는
  스펙만 있고 정식 상품명이 없어서, title/h1/JSON-LD name에 쓸 정식명은
  빌드 시점에 시트를 fetch해서 가져온다. 단 이건 가격과 달리 자주 안
  바뀌는 값이라 정적으로 박아도 괜찮다고 판단함. → 이것 때문에 워크플로우
  트리거에 "매일 1회 스케줄"이 필요함 (시트에 신상품이 추가되는 경우
  코드 push 없이도 반영되게).
- **키워드 매칭 방식을 쓴 이유**: 기존 `coin-detail.html?name=...`은
  시트의 정확한 전체 상품명과 1:1 매칭이라 상품명이 조금만 바뀌어도
  깨진다. 새 방식은 `coin-descriptions.js`/`IMAGE_MAP`과 동일한 "이름에
  특정 키워드가 포함되는지"로 매칭해서 더 안전하다. 다만 시트 상품명이
  키워드를 전혀 포함하지 않게 적히면(예: 정식명에 코인 별명이 하나도
  안 들어간 경우) 조용히 skip되니, 실전 실행 후 skip 목록을 꼭 확인할 것.
- **PAT는 일반 push와 workflow 파일 push의 필요 권한이 다르다**:
  `.github/workflows/` 안의 파일을 push하려면 PAT에 별도로
  `Workflows: Read and write` 권한이 있어야 한다. Fine-grained PAT는
  발급 후 권한을 못 바꾸므로, 다음에 또 workflow 파일을 고칠 일이
  있으면 이 권한이 포함된 토큰인지 미리 확인하고 받을 것.
- **GitHub Actions 푸시는 `GITHUB_TOKEN` 사용**: 워크플로우 자체의
  자동 커밋&푸시는 사용자 PAT가 아니라 GitHub Actions 기본 제공 토큰
  (`permissions: contents: write`로 설정)을 쓰므로, 사용자가 PAT를
  매번 발급할 필요는 없다. PAT는 이번처럼 Claude가 직접 새 파일을
  push할 때만 필요.
- **앞면/뒷면 기준**: 주제 그림(동물·건물 등) = 앞면(`.png`), 인물
  초상·국가 문장 = 뒷면(`-back.png`)
- **스펙 필드 구조**: `year`, `mint`, `country`, `purity`, `weight`,
  `diameter`, `thickness`(선택), `condition`, `verified`
- **신규 상품 추가/삭제 시**: `coin-descriptions.js` + `products.js`의
  `IMAGE_MAP` + 시트 행 3곳을 함께 관리. SSG 도입 후에는 이 셋이
  자동으로 `coin-{slug}.html`까지 만들어주므로, 사람이 직접 만들 파일이
  하나 줄었다 (단, 시트 행 추가는 여전히 사람이 수동으로 해야 함 —
  시트 쓰기는 git으로 자동화 불가).
