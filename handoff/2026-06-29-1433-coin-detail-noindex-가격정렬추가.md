## 2. 현재 상태

### (1) coin-detail.html — 정리 보류, noindex만 추가 (확인 완료)
- 검토 결과: 실제 fallback 발동 사례는 0건(코인 20개 전부 slug 매칭 성공)이라
  사실상 죽은 경로이나, **새 코인 추가 시 IMAGE_MAP 키워드 매칭이 실패하는
  경우의 안전망**으로 동작하므로 페이지 자체는 삭제하지 않기로 결정.
- 다만 `robots.txt`/메타태그 어디에도 색인 차단 장치가 없어 우연히 인덱싱될
  위험이 있었음 (확인 완료: sitemap.xml에는 없었으나 noindex 메타태그도 없는
  상태였음).
- 수정: `coin-detail.html` 8번째 줄(terms.html/privacy.html과 동일 위치)에
  `<meta name="robots" content="noindex, follow">` 추가. 커밋 `017b0c6`.

### (2) coins.html — 가격 낮은순/높은순 정렬 기능 추가 (확인 완료)
- 구현 방식: `data-premium`(시트 프리미엄율) 값은 모든 코인에 공통으로 곱해지는
  `krwPrice`(국제 시세)와 무관하게 가격의 상대적 순서를 그대로 반영하므로,
  시세 fetch 완료 여부와 무관하게 `data-premium` 기준으로 즉시 정렬 가능함을
  이용함.
- UI: 기존 브랜드별 드롭다운(`.ptab-brand-wrap`) 옆에 정렬 드롭다운
  (`#ptab-sort-wrap` / 기본순·가격낮은순·가격높은순) 추가. 기존 탭 필터와
  독립적으로 동시 적용 가능 (display 토글과 DOM 순서 재배치가 분리되어 있어
  충돌 없음, 확인 완료).
- 적용 범위: (a) 일반 진입(검색 쿼리 없음, SSG 정적 카드) — `applySortOrder()`
  가 DOM 카드를 `data-premium` 기준 재배치. (b) 검색 결과(`renderProducts()`)
  — 정렬 기준(`currentSort`)을 그대로 반영해 처음부터 정렬된 상태로 그림.
- SSG(`generate-coin-pages.js`)는 건드리지 않음. 정렬은 순수 클라이언트
  DOM 재배열이라 크롤러용 최초 HTML(재고우선+order)에는 영향 없음 — SEO
  안전 (의도된 설계).
- 커밋 `3846cfd` (feat: coins.html 가격 낮은순/높은순 정렬 기능 추가).

### (3) 버그 발견 및 수정: 검색 결과에서 "기본순" 복원 안 되는 문제 (확인 완료)
- 위 (2) 구현 직후 코드 검증 과정에서 발견. `renderProducts()`가 정렬 복원용
  기준값(`origIndex`)을 부여할 때, **이미 `currentSort`로 정렬된 순서**를
  그대로 기준값으로 써버려서, 검색 결과 화면에서 가격순 적용 후 "기본순"을
  눌러도 진짜 기본순(재고우선+order)이 아니라 직전 가격순서가 그대로
  남아있는 버그였음 (시뮬레이션으로 재현·검증 완료).
- 일반 진입(검색 없음) 경로는 영향 없음 — SSG 카드는 처음부터 한 번만
  origIndex가 부여되므로 문제없었음.
- 수정: `sorted`(현재 정렬 적용 결과)와 별개로 "재고우선+order" 기준 순위를
  한 번 더 계산(`baseRank`)해서, `currentSort`와 무관하게 항상 같은 기준값을
  `origIndex`로 부여하도록 변경. 수정 후 시뮬레이션으로 정상 복원 확인 완료.
- 커밋 `d519d07` (fix: 검색 결과에서 정렬 후 기본순 복원 시 원래 순서로
  안 돌아가는 버그 수정).

## 5. 참고 사항

- **`data-premium` 기준 클라이언트 정렬 패턴**: 이번에 확립된 패턴 —
  `krwPrice`(국제 시세)는 모든 코인에 동일하게 곱해지는 양수 상수이므로,
  가격의 상대적 순서는 `premium` 값만으로 결정된다. 시세 fetch 타이밍과
  무관하게 즉시 정렬 가능하다는 점을 활용함. 향후 다른 가격 기반 기능
  (예: 가격대 필터)에도 동일 원리 적용 가능.
- **origIndex 패턴 주의사항**: "원래 순서로 복원" 기능을 만들 때, 복원
  기준값을 "현재 화면에 그려진 순서"로 매기면 안 되고 항상 "정렬 기준과
  무관한 고정 기준(이번 경우 재고우선+order)"으로 별도 계산해서 매겨야
  한다. 화면 순서를 그대로 기준값으로 쓰면 한 번이라도 다른 정렬을 거친
  뒤에는 복원이 무의미해지는 버그가 재발하기 쉬움 — 이번처럼 정렬/필터
  기능을 추가할 때마다 함께 점검할 것.
- **검색 경로(`renderProducts()`)와 일반 진입 경로(SSG 카드)는 별개 코드
  경로**: 둘 다 같은 `.product-card` DOM을 다루지만 그려지는 시점과 방식이
  다르므로, 정렬/필터 등 신규 기능을 추가할 때는 항상 두 경로 모두에서
  동작하는지 따로 확인해야 함 (이번 버그도 검색 경로에서만 발생했었음).
