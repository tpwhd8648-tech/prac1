# SSG 초기 설계 대조 검증 및 의사결정 인수인계 문서

작성일: 2026-06-23

## 1. 목표

SSG 작업 초기 설계 논의(과거 대화 기록)에 나온 체크리스트를 현재 코드와
하나씩 대조 검증하고, 설계 당시 합의와 실제 구현 사이에 발견된 2가지
차이점(빌드 주기, id/name 파라미터)이 실제로 문제가 되는지 판단한다.

## 2. 현재 상태

**검증 완료 — 과거 설계 논의 체크리스트 전체 대조 (코드 직접 확인)**:

| 항목 | 상태 |
|---|---|
| 🔴 1-1 GitHub Actions 워크플로우 | ✅ 완료 — `.github/workflows/generate-coin-pages.yml` 존재 |
| 🔴 1-2 코인별 정적 HTML (슬러그 기반) | ✅ 완료 — 설계 당시 19개였는데, 이후 세션에서 코인 1개(레이디저스티스) 추가돼 현재 20개 |
| 🔴 1-3 coins.html 카드 링크 교체 | ✅ 완료 — `getSlugForProduct()` + `linkTemplate` |
| 🟡 canonical 코인별 분리 | ✅ 완료 |
| 🟡 og:title/og:image 코인별 분리 | ✅ 완료 |
| 🟡 JSON-LD 정적 포함 + Product 스키마 | ✅ 완료 (price/availability는 이전 세션에서 추가 완료) |
| 🟡 sitemap.xml 코인 19개(현재 20개) URL | ✅ 완료 |
| 🟡 mypage.html og/canonical/favicon | ✅ 완료 |

**발견된 차이점 2가지 — 판단 완료**:

1. **빌드 주기**: 설계 당시 "새벽 6시 KST"로 합의했으나, 실제 워크플로우는
   `cron: '0 15 * * *'`(UTC) = **KST 00:00**으로 구현됨.
   → **판단: 의미 없는 차이, 수정 불필요**. SEO 목적은 "하루 1회,
   24시간 이내 시세"이고 어느 시각이든 본질은 동일함. 현재 00:00 KST
   그대로 유지하기로 결정.

2. **coin-detail.html 쿼리파라미터**: 설계 당시 "기존 `?id=메이플` 방식
   링크"라고 언급됐으나, 실제 `coin-detail.html` 코드(539~540행)는
   처음부터 `URLSearchParams`로 **`name`** 파라미터만 읽음 — `id`를
   읽는 코드는 어디에도 없음.
   → **판단: SSG 작업이 `id`→`name`으로 바꿔서 링크를 깨뜨린 게
   아니라, 원래부터(SSG 작업 이전부터) `name` 방식이었던 것으로 보임.**
   단, shallow clone이라 git 히스토리가 6개 커밋밖에 안 남아있어 100%
   확정은 불가. **미확인 사항**: 사용자가 카카오톡/SNS/블로그에
   `?id=...` 형태로 공유한 링크가 실제로 있었는지는 사용자만 알 수
   있음 — 질문했으나 이번 세션에서 답변받지 못함.

**이번 세션에서 새로 결정된 것**:
- `coin-detail.html` 자체의 JSON-LD 정적 전환 작업은 **"보류"가 아니라
  "안 하기로 결정"** — fallback 전용 페이지라 비용 대비 효과가 낮다는
  사용자 판단. (이전 핸드오프에 🔵 보류로 적혀 있던 항목을 종결 처리)

## 5. 참고 사항

- **이번 세션의 출발점**: 사용자가 과거 SSG 설계 초기 논의 대화 기록을
  문서로 붙여넣고, 거기 나온 체크리스트(1-1~2-1)가 전부 완료됐는지
  재검증 요청. 모두 코드 직접 확인으로 완료 확인됨.
- **shallow clone의 한계**: 이 작업 환경은 `--depth 1`로 클론하기
  때문에 git 히스토리가 최근 몇 개 커밋만 보임. "예전에 이 코드가
  어땠는지"를 git log로 직접 확인하기 어려운 경우가 있음 — 이번
  세션의 id/name 파라미터 판단처럼, 과거 상태는 git 히스토리 대신
  현재 코드 동작과 사용자 기억에 의존해야 할 때가 있다는 점을
  인지하고 접근할 것.
- **빌드 주기/파라미터 명명 같은 "설계 논의 당시의 디테일"과 "실제
  구현"이 다를 수 있음**: 설계 논의는 보통 구현 전 추상적인 합의이고,
  실제 코딩 단계에서 합리적인 이유로 살짝 달라지는 경우가 있음(이번처럼
  빌드 시각). 이런 차이를 발견했을 때 무조건 "설계안대로 고쳐야 한다"가
  아니라, 차이가 실질적 영향이 있는지부터 판단하고 사용자에게 알리는
  방식이 더 효율적이었음 — 이번 세션의 접근 방식을 다음에도 유지할 것.
- **SSG 전체 구조 — 최종 확정 상태**: 코인 정적 페이지 생성(Actions,
  20개) + coins.html 링크 연결 + 화면 가격 연동 + JSON-LD
  canonical/og/price/availability 전부 완료. `coin-detail.html`은
  fallback 전용으로 유지, 추가 작업 없음. 신규 코인 추가 시:
  `coin-descriptions.js` + `products.js`의 IMAGE_MAP + 시트 행 3곳
  추가하면 다음 Actions 실행 시 자동으로 `coin-{slug}.html`까지
  생성됨 (가격/재고 포함).
- **앞면/뒷면 기준**: 주제 그림(동물·건물 등) = 앞면(`.png`), 인물
  초상·국가 문장 = 뒷면(`-back.png`)
- **스펙 필드 구조**: `year`, `mint`, `country`, `purity`, `weight`,
  `diameter`, `thickness`(선택), `condition`, `verified`
